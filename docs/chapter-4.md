# 第4章 UML（統一モデリング言語）

UMLはプログラム仕様を正確に記述する言語ではなく、ノイズを省いて人間同士の設計意図を伝えるコミュニケーション記法である。クラス図・インスタンス図・シーケンス図はそれぞれ異なる切り口で構造と委譲の様子を可視化する道具として使われる。

## 章の主張

UMLはプログラム仕様を正確に記述する言語ではなく、ノイズを省いて人間同士の設計意図を伝えるコミュニケーション記法である。クラス図・インスタンス図・シーケンス図はそれぞれ異なる切り口で構造と委譲の様子を可視化する道具として使われる。

## 前提

- 読み書きする全員が矢印の向き・ひし形の種別・依存の向きの定義を共有している
- UMLをコードと同期しつつ保守し続けることを目的としていない
- シーケンス図の有効性はTell, Don't Ask型の委譲設計が成立していることを前提にしている
- 集約・コンポジションの区別が「コードに現れない」という主張は、Java等の主流オブジェクト指向言語を前提にしている

## 実務上の効きどころ

- 設計レビューでコードを読ませる前に、依存の向き・所有関係の意図を一枚の図で共有する
- 集約／コンポジションの区別を図に明示することで、オブジェクトの寿命・廃棄責任がどこにあるかを言語化できる
- シーケンス図はアルゴリズムの網羅的記述ではなく、オブジェクト間の委譲の構造概要を伝える用途に絞って使う
- ツール不要・ホワイトボード手書きで即使える

## 弱点と例外

- 記法の合意がないチームでは、矢印の誤読がむしろ設計の混乱を固定化する
- 「ノイズを省くほど価値が上がる」は単調増加の主張であり、省きすぎると受け手が情報を補完しなければならなくなる下限がある
- 集約・コンポジションの「コードに現れない」という主張はRust等の言語では成立しない（本文は注釈で示唆するにとどまり断言を自己修正していない）
- Tell, Don't Askが成立しないコードベースでシーケンス図を使おうとすると何が起きるかは本章に書かれていない
- 「緻密なUML＝幻想」の断言は、コード生成ツールやMDAの実用例の検討なしに普及しなかった事実だけを根拠にしており、論拠が薄い

## 議論用の問い

1. 「どこまで描くか」の判断基準をチームでどう合意するか。粒度の上下限が本章に書かれていない以上、現場では何を基準にして決めているか。
2. 集約とコンポジションの区別は、自分たちのコードベース・使用言語において実際に図の上で使い分ける価値があるか。
3. UMLを「都度使い捨て」にするアプローチと「ドキュメントとして保守する」アプローチのどちらを自分たちのチームは取っており、それは本章の立場と整合しているか。

== 理解の視点

**拾った主張**
- UML（Unified Modeling Language）とは、ソフトウェア設計の図の書き方を標準化したもの。「言語」とは、ルールにのとったコミュニケーション手段のことで、図であっても正確なきまりがあれば言語になる。
- クラス図では長方形がクラス・インターフェース、フォルダ形がパッケージを表す。インターフェースは小さな円で代用することもある。
- 関係線の種類：何も付かない線は参照関係、矢印は依存の向き、白抜き矢印は継承（パッケージ内でも省略しない）。継承は他の依存と異なり、組み換えできない強い結合。
- 矢印（依存の向き）の意味はデータや処理の流れではなく、常に「依存の方向」を表す。
- 数量関係は線の端に数字や `*` で表す。`1...*` は1以上で上限なし。
- アグリゲーション（集約・白抜きひし形）：所有者は既存オブジェクトへの参照を保持。所有者がなくなっても参照されたオブジェクトの生存は無関係。共有所持も可能。
- コンポジション（黒塗りひし形）：所有者がオブジェクトを専有。所有者がなくなると子の存在意義も消滅。子にとって親は唯一かつ絶対。
- 集約・コンポジションはコード上の宣言では区別がつかない。違いは「意図」と実際の動作にある。
- インスタンス図（オブジェクト図）：プログラム実行時の特定時点のオブジェクト実体の状態を表す。抽象クラス・インターフェースは含まれない。インスタンス化されていないクラスも含まれない。
- シーケンス図：インスタンス図の時間軸版。上から下に時間が流れ、オブジェクト間のメソッド呼び出しと応答を矢印で表す。分岐・ループなどのアルゴリズムを網羅的には表現できない。
- シーケンス図の目的は処理手順ではなく、オブジェクトの委譲の様子（構造の概要）を見ること。
- UMLはプログラミング言語ではない。UMLの価値はノイズとなる情報を省略することにある。緻密なUMLでプログラムを代替しようとした試みは失敗した。UMLは対人間用コミュニケーション言語である。

**気にした論点**
- 「Tell, Don't Ask」という用語がシーケンス図の説明中に突然出てくるが、定義がない。前の章で定義されている前提で書かれているようだが、この章だけ読むと意味が掴みにくい。
- 「集約とコンポジションの違いはコード上には現れない、意図の違いでしかない」と言いながら、「実際の動きの違いになる」とも書かれていて、この2文の関係が少し追いにくかった。
- 注釈で「ドメイン駆動設計の"集約"とは別物」と書かれているが、どう違うのかはここでは触れられていない。

**結論**

この章は、UMLの主要な図（クラス図・インスタンス図・シーケンス図）の読み書きのルールを説明しつつ、UMLは正確さではなく人間の理解のためにノイズを省く道具だという立場を示した話。

**残した保留**
- 「Tell, Don't Ask」が何を指すかがわからないまま。
- 集約（UML）とドメイン駆動設計における集約の「共通点も相違点もある」の具体的中身が不明。
- シーケンス図でgetterが絡む場面が「あまり向いていない」理由が直感的にはまだよくわからない。

== 実務の視点

**拾った主張**
- **UMLは対人間用コミュニケーション言語である**（適用条件：共同開発メンバーへの設計説明、自分の理解整理）
- **パッケージ図とクラス図は組み合わせて使うべき**（適用条件：パッケージ間の依存関係とクラスの概念構造を同時に見たいとき）
- **矢印はデータの流れではなく依存の向きを表す**（適用条件：依存方向を正確に伝えたいとき。パッケージ境界をまたぐ場合に特に有効）
- **シーケンス図は「委譲の様子・構造の概要」を表すもの**で、アルゴリズム・分岐・ループの網羅的表現には向かない（適用条件：オブジェクト間の責務の流れを伝えたいとき限定）
- **集約とコンポジションの区別は意図の違いであって、コード宣言には差が出ない**（適用条件：設計の意図を読み手に伝えるため、図の段階で明示したいとき）
- **緻密なUMLをプログラミングの肩代わりにしようとするのは幻想**と本書は断言している

**気にした論点**

使える場面：
- 設計レビューの場で、コードを読ませる前に構造の意図を共有するツールとして有効
- 集約／コンポジションの区別は、オブジェクトの寿命設計を言語化するのに使える（特に削除・廃棄の責任がどこにあるかを確認するとき）

使えない場面・注意点：
- 矢印の向き（依存方向）を誤解したまま描かれたUMLは、むしろ設計の混乱を固定化する。読み書きのルールを全員が共有していないと逆効果
- シーケンス図でif/elseを書こうとして破綻するパターンは現場でよく起きる。本書が「アルゴリズムを表現しない」と明示しているのは重要な制限事項
- 「ノイズになる情報を省けば省くほど価値が高い」という主張は、何を省くかの判断基準を書いている人の力量に依存する

導入コスト：
- UMLツールを揃えなくても、ホワイトボードや手書きで十分機能すると読める
- ただし、チーム内で記法の合意（特に矢印の向き・ひし形の区別）がないと個人差で誤読が起きる

既存手法との差分：
- 本書はUMLを「正確性」ではなく「伝わりやすさ」の道具として定義している。これは「UMLをドキュメントとして保守する」アプローチとは相容れない

**結論**

設計レビューや新規メンバーへの構造説明で、コード全部を読ませる前に「依存の向き」と「集約／コンポジション」だけ図に落とす用途なら、今日から使える。

**残した保留**
- 「パッケージ内では矢印を省略してよい」という主張は、パッケージの粒度設計に依存する。パッケージが大きければ内部の相互依存も問題になるはずで、適用条件の境界が不明
- 「緻密なUMLは幻想」と言いながら、どのくらいの粒度が適切かの具体的な判断基準は本章に書かれていない。現場で「どこまで描くか」で時間を溶かすリスクが残る
- シーケンス図が「Tell, Don't Ask委譲を前提にしている」という記述があるが、その前提が成立しないコードベースでシーケンス図を描こうとしたとき何が起きるかは書かれていない

== 批評の視点

**拾った主張**

- **「UMLはつまらない誤解を避けるためのものだ」**
  → UML仕様自体が巨大で、矢印の向きや集約・コンポジションの解釈は人によって揺れる。誤解を生みにくい記法が、新たな誤解を生む場合に崩れる。

- **「集約とコンポジションは意図の違いでしかない」**（注1で自ら「言い切れない」と後退している）
  → 注1の但し書きで主張を自己修正しているが、本文の断言と矛盾している。条件が明示されていない主張として成立していない。

- **「継承はそうそう組み換えできるものではない。概念的なis-a関係が変わるなんておかしい」**
  → 設計の経験則であって論理的必然ではない。リファクタリングで継承関係を変えることは実務で起きる。「おかしい」という価値判断が論拠になっている。

- **「ほとんどの場合が1->* と *->1 あるいは *-* になる」**
  → 「ほとんど」の根拠がない。対象ドメイン・設計スタイル・チームの習慣によって分布は異なる。

- **「シーケンス図はアルゴリズムを表現しない」**（UML2が普及しなかったことを根拠として挙げている）
  → 「普及しなかった」理由が「技術的に誤りだったから」なのか「複雑すぎたから」なのかが示されていない。普及しなかった事実は、記述能力がないことの証明にはならない。

- **「ノイズを省けば省くほどUMLの価値は高くなる」**
  → 省きすぎると情報が失われ、受け手が補完しなければならなくなる。単調増加の主張として過剰一般化。

- **「緻密なUMLはプログラムコードよりはるかに無駄が多く複雑になる」**
  → 幻想だったと断言しているが、その根拠が示されていない。UMLからのコード生成ツールが存在することへの言及がない。

**気にした論点**

前提が明示されていない点：
- 集約・コンポジションの区別が「意図の違いでしかない」という主張は、どのプログラミング言語を前提にするかによって成否が変わる。注1で認めているにもかかわらず、本文の断定を修正していない。

論理の飛躍：
- 「緻密なUMLはプログラムコードよりも無駄が多い」→「だからUMLはコミュニケーション用に限る」という結論は、正確さの代償と用途の整理を混同している。緻密さが不向きなことと、コミュニケーション専用であることは別の命題。

根拠の薄さ：
- 「ほとんどの場合が1->*か*->1」は経験則に過ぎず、裏づけがない。
- UML2が普及しなかった理由が「記述能力が問題だったから」とは証明されていない。

仕様外の推奨：
- クラス図とパッケージ図を組み合わせることについて「UML仕様上は別」と認めながら「ぜひ使おう」と断言している。仕様からの逸脱を推奨する根拠が「視認性」だけでは弱い。チームやtoolの対応状況に依存する話が抜けている。

**結論**

UMLが「プログラミング言語ではなく、人間間のコミュニケーション用の記法である」という基本的な位置づけについては、論理的に一貫しており、最も根拠が支えられている部分である。

**残した保留**
- 集約・コンポジションの意図の違いが「コードに現れない」という主張は、言語依存の例外（Rust等）を注1で示唆するにとどまる。どこまでの言語範囲で成立する主張なのか検証が必要。
- 「緻密なUML＝幻想」という主張は、MDA（Model Driven Architecture）やコード生成ツールの実用例を検討すれば覆る可能性がある。
- シーケンス図が「アルゴリズムを表現しない」というのは設計上の選択なのか、根本的な限界なのか区別されていない。UML2の失敗との因果関係が未検証。
- 「ノイズを省くほど価値が上がる」という単調増加の主張は、省略の下限（最低限残すべき情報量）について何も言っていない。
