# 第5章 オブジェクト指向 SOLID

変わりやすい仕様と変わりにくい本質を分離し、依存の向きを呼び出し構造ではなく安定度（抽象度）で決めることがクラス設計の核心である。SOLID の5原則はこの一軸に沿って接続されており、DIP によってクリーンアーキテクチャの依存逆転が成立する根拠が与えられる。

## 章の主張

変わりやすい仕様と変わりにくい本質を分離し、依存の向きを呼び出し構造ではなく安定度（抽象度）で決めることがクラス設計の核心である。SOLID の5原則はこの一軸に沿って接続されており、DIP によってクリーンアーキテクチャの依存逆転が成立する根拠が与えられる。

## 前提

- 将来の変化がある程度予測できる、または変化点の見当がつけられる
- 長寿命が見込まれるモジュールが対象である（一時スクリプトへの適用は対象外）
- 設計者が「交換したくなる単位」を想像できる経験・文脈を持っている

## 実務上の効きどころ

| 原則 | 使いどころ |
|------|-----------|
| SRP・ISP | 差し替えが起きやすい結合点。利用者ごとに見せる顔を絞ることで、拡張時に既存コードへの変更を防ぐ |
| OCP | 長生きすると判断した箇所への先行投資。スモールスタートから段階的に導入できる |
| LSP | 継承・カスタマイズを使う全員が守らないと、呼び出し側全体に修正が波及する隠れバグを生む |
| DIP | 技術レイヤーとビジネスロジックの依存方向を設計する場面。導入コストは高いが長期保守で回収できる |

## 弱点と例外

- **SRP の恣意性**: 「責務の単位は設計者の想像で決める」と本文が認めているため、同一クラスへの正反対の判断が同じ論法で正当化できる。客観的な検証条件が示されていない。
- **OCP の予測依存**: 「本質の見立て」が外れた場合への対処が述べられていない。変化点の予測は経験依存で、手順書化が難しい。
- **LSP の「黙って動く」**: ゼロ除算を 100% に置き換えることを推奨する一方、防衛的コードを批判する箇所と共存しており、許容範囲の境界条件が明示されていない。セキュリティ重視の文脈では崩れる可能性がある。
- **コラムの論理矛盾**: 本文で「SOLIDはOO専用でない」と述べながら、コラムでは「SOLIDに則していないものをOOと呼ぶのは難しい」と主張しており、論理が整合しない。
- **クリーンアーキテクチャの単純化**: 「DIPを適用した応用例にすぎない」という断定は、境界・ユースケース・エンティティの独立性など他の設計判断の存在を捨象している。

## 議論用の問い

1. 「責務の単位は恣意的に決める」と認めておきながら、SOLIDが「守るべき原則」として成立するのはなぜか。設計の指針と原則は同じ強さの規範なのか。
2. LSP における「黙って動く（ゼロ除算を 100% に置換する）」推奨と、他箇所での「防衛的コードの批判」は、どう使い分けるべきか。境界条件は何か。
3. SOLIDが「OO専用でない」ならば、関数型・手続き型での適用は自分たちのコードベースでどう翻訳できるか。翻訳できない部分はどこか。

== 理解の視点

**拾った主張**
- **SOLID** とは、SRP・OCP・LSP・ISP・DIP の5原則の頭文字をとった略語で、クラス設計の粒度に関する原則
- SOLID は本質的にオブジェクト指向専用ではなく、ソフトウェア設計全般に応用できる考え方
- **SRP（単一責任）**: 1クラス＝1責務。「独立して交換したくなる単位」が責務の定義。責務の線引きは機械的に決まらず、将来の保守への想像から恣意的に決める
- **OCP（開放閉鎖）**: 仕様が変わる部分と変わらない本質を分離する。本質側は変更に対してクローズド、仕様の追加に対してオープンな作りにする。FizzBuzz の例では「倍数かどうか」が仕様で、「ルールを累積適用する」という振る舞いが本質
- **LSP（リスコフの置換）**: 派生クラスは基底クラスの使い方をすべてカバーしなければならない。事前条件（入力制約）を増やすのも、事後条件（出力範囲）を逸脱するのも違反。ソフトウェアのバージョンアップにおける下位互換性の話でもある
- **ISP（インターフェース分離）**: インターフェースは「利用者が認識する部分概念」の最小単位。クライアントコードが異なるパッケージにあるとき、クラスは複数の関心を持っているため、利用者ごとに小さなインターフェースを切って接続する
- **DIP（依存性逆転）**: 依存の向きは、呼び出し構造の上位・下位ではなく安定度（抽象度）で決める。安定した抽象に向かって依存させることで、制御の流れと依存の向きが逆転する
- DIP はクリーンアーキテクチャの謎（技術実装レイヤーがビジネスロジックレイヤーに依存できる理由）の答えにあたる

**気にした論点**
- SRP における「責務の単位」は将来への想像で恣意的に決めると述べているが、「恣意的」にしか決められないことが本当にルールとして機能するのかが気になった
- OCP の FizzBuzz で「本質の洗練」をどこで止めるかについて、「これ以上分けてはいけない」という判断基準がテキスト上で明示されていない
- LSP の「黙って動く（total=0 のとき 100% とする）」が「良くないプログラミングに見えるがそれより互換性維持が重要」という判断の根拠が直感的に受け入れにくかった
- ISP と SRP の違いが「クラス版 vs インターフェース版」と説明されているが、具体的にどちらで悩むべきか、ISP を守るとき SRP はどこに行くのかが追いにくかった
- DIP の説明で「構造化プログラミング vs オブジェクト指向」という対比が何度か出てくるが、構造化プログラミングの定義がテキスト内に明示されていない

**結論**

第5章は、クラス・インターフェース設計の5つの原則（SOLID）を通じて、「変わりやすい仕様と変わりにくい本質を分離し、依存の向きを安定度に従わせる」ことがオブジェクト指向設計の核心であると述べ、それがクリーンアーキテクチャの依存方向の謎を解く鍵になるという章。

**残した保留**
- 「責務の単位は恣意的に決める」という話と「原則に則している」という話が、どう整合するのかがまだ腑に落ちていない
- DIP によって「依存の向きが制御の向きと逆転する」のが「自然に起こる」と言われているが、「自然に」という感覚をまだつかめていない。具体的にどういう設計手順を踏むと逆転するのかを追い切れていない
- コラム部分（p149）で「SOLID に則していないとオブジェクト指向とは言いにくい」という筆者意見が出るが、これは本文の原則説明と同じ強さの主張なのか、個人見解なのかの区別を保留している

== 実務の視点

**拾った主張**
- **SRP「交換の単位で責務を判断する」** — 教科書的な「ひとつの責務」より、「いつ、誰が、どの部分を差し替えたくなるか」で判断する方が実際に動く。適用条件：差し替えニーズが将来予測できるシステム
- **OCP「本質と仕様を別パッケージに分ける」** — 長生きするとわかった箇所だけ早めにやる、という選択的適用の指針が書いてある。スモールスタートから段階的に導入できる。適用条件：将来の仕様変更が見えている箇所、寿命が長いモジュール
- **LSP「呼び出し側の使い方仕様を守る」** — 「制約を追加してはいけない」が実装継承でもインターフェース実装でも等しく成立する点は現場で見落としやすい。適用条件：既存クラスを拡張・カスタマイズするとき全般
- **ISP「利用者の数だけインターフェースを用意する」** — クラスをひとつに保ちつつ、利用者ごとに見せる顔を変える。Bluetooth拡張の例が示すように、これをやっておくと後の拡張で既存コードに触らずに済む。適用条件：複数の異なる文脈から同一クラスを使うとき
- **DIP「依存の向きを呼び出し構造ではなく安定度で決める」** — クリーンアーキテクチャが「なぜ技術レイヤーがビジネスロジックに依存できるのか」の根拠がここで回収される。導入コスト高だが、長期保守では回収できる

**気にした論点**

使える場面 / 使えない場面：
- SRP・OCPは「長生きするとわかった部分」に先行投資として入れる判断ができる。一時スクリプトに適用するのは明確にやりすぎ
- LSP違反の実例（ゼロ除算バグ）は現場でも起きる。継承を使うチーム全員が意識しないと片手落ちになる
- FizzBuzzのOCP抽象モデル版は「最初から正しく予測できた場合」の模範解答。現実は事後リファクタリングのほうが多い
- DIPの「制御の向きと依存の向きが逆になってよい」は構造化プログラミング出身者には直感と逆。教育コストが大きい

**結論**

「長生きするとわかった箇所」にOCP・DIPを、「差し替えが起きやすい結合点」にSRP・ISPを選択的に当てる判断基準として使えば、設計コストの投資対効果が見えてくる。

**残した保留**
- 「変化する仕様の予測」を組織的にどう訓練・共有するか。個人の経験に依存したままでは属人的スキルで終わる
- FizzBuzzの抽象モデル版は「最初から設計した場合」の話。既存のベタ書きコードをこの形にリファクタリングするときのコストと安全な手順は本文に書かれていない
- SOLIDが「オブジェクト指向限定でない」と筆者は述べているが、関数型・手続き型での具体的な適用例がここでは示されていない

== 批評の視点

**拾った主張（と崩れる場面）**

- **SRP「クラスと責務はぴったり1:1対応すべし」** → 「責務」の定義が設計者の主観次第で正反対の判断が正当化できる。客観的に検証できる条件が示されていない
- **OCP「変化する仕様を予測して本質を分離すれば変更なしに拡張できる」** → 予測が外れた場合への対処がない。「本質」の見立ても恣意的
- **LSP「異常を隠して黙って動くべき」** → ゼロ除算を黙って100%に置換することは障害検知・セキュリティ観点から問題になりうる。「下位互換」と「誤入力の黙認」を同一視している
- **コラム「SOLIDに則していないものをOOと呼ぶのは難しい」** → 直前で「SOLIDはOO専用でない」と述べており論理が整合しない
- **「クリーンアーキテクチャはDIPの応用例にすぎない」** → 境界・ユースケース・エンティティの独立性など他の設計判断の存在を無視した過剰単純化

**気にした論点**
- DBドライバは「読み書きを一緒にすべき」、ArticleOperationは「入稿と購読を分けるべき」という判断の違いは、同じ論法で逆の結論も導けてしまう。判断基準の一貫性が確認できない
- コラムの「現在も生き残っている技法はSOLIDに則した特徴を持っている」は生存者バイアスへの配慮がない

**結論**

SOLIDの各原則を「依存方向の安定度」という一貫した軸で接続し、DIPによってクリーンアーキテクチャの依存逆転が自然発生することを示す論理は、コード例を通じて比較的十分に支えられている。

**残した保留**
- 「責務の恣意性」を認めた上でなぜSOLIDが「守るべき原則」として成立するのか、その正当化の根拠が本章では与えられていない
- LSPの「黙って動く」が崩れる境界条件（セキュリティ重視のシステム等）が明示されていない
- SOLIDが「OO固有でない」ならば、OO以外のパラダイムで適用した場合に同等の効果が得られるかどうかは未検証
