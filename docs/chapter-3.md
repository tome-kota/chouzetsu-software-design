# 第3章 オブジェクト指向

この章の中核は、インターフェース（機能する抽象）を使うことで、上位パッケージを下位の詳細から切り離せることにある。カプセル化・多態性・継承はいずれも、この「上位構造を安定させる」目的に向かう手段である。

## 章の主張

インターフェース（機能する抽象）を使うことで、上位パッケージを下位の詳細から切り離せる。カプセル化・多態性・継承はいずれも、この「上位構造を安定させる」目的に向かう手段である。

## 前提

- 問題になるのはパッケージ境界を越える依存であり、パッケージ内の密結合は許容される。
- 抽象は具象の分析から帰納的に発見するものであり、先行した抽象立案は機能しない。
- 「機能する抽象」の主張は、型システムのある言語環境を主な前提としている（ダックタイピングでの成立は部分的）。

## 実務上の効きどころ

- DB や UI の詳細が未確定な初期フェーズで、インターフェースを先に書いて上位パッケージを先行安定させる。
- if/switch の切り替え分岐が増えてきたタイミングで、多態性（NullObject パターン等）に置き換える。
- 外部パッケージとの境界を越えるコードでデメテルの法則を適用し、getter 連鎖を委譲に変える。

## 弱点と例外

- 「OOP を定義できない」と言いながら3特徴を列挙する移行が無断で行われており、出発点の論理的厳密さが弱い。
- 「最小の抽象が最高の拡張性を持つ」は根拠となる事例も研究も示されておらず、主張の強度に見合う証拠がない。
- C の関数ポインタ・不透明ポインタでも「呼び出し側が実装を知らずに動作する」は実現でき、構造化と OOP の「決定的な違い」が揺らぐ。
- デメテルの法則における「抽象度が同じレベル」の判断基準が主観的で、チーム間で適用が割れやすい。
- 汎化のタイミング（具象を何個観察すれば帰納できるか）の基準が示されておらず、締め切りのある現場への翻訳が未解決。

## 議論用の問い

1. 「抽象度が同じレベルのメソッドチェーンは問題ない」という判断を、実際のコードレビューでどう運用するか。主観を排した基準は作れるか。
2. 「具象の分析から帰納的に抽象を発見せよ」という原則は、DDD のようにドメインモデルを事前に議論するアプローチとどう折り合いをつけるか。
3. インターフェースを増やすことのオーバーヘッド（ファイル数・読むコスト・チーム習熟コスト）と、上位パッケージの安定性が得られるバランスは、どんな規模・チームで成り立つか。

== 理解の視点

**拾った主張**
- **オブジェクト指向に定義はない**：「モノのイメージでプログラミングをうまくやろう」としか言えない。明確な定義から発展したパラダイムではない。
- **オブジェクト指向は道具**：単独で価値を持つパラダイムではなく、より良いパッケージングに役立てることに価値がある。
- **否定による定義**：①現実の再現ではない、②クラス構文の文法ではない、③状態管理の仕組みではない、④手続き型プログラミングのことではない。
- **カプセル化**：相互に関連性の高い変数と処理を1つのオブジェクトに閉じ込める。凝集度の最小単位。詳細を隠蔽し、関心を分離する境界線として機能する。
- **責務の委譲**：オブジェクトは、自身の責務を浅くして内部詳細を別オブジェクトに委ねる。メソッドチェーンでカプセルの中を掘り出す行為はデメテルの法則違反。
- **多態性 = プラグイン**：呼び出し側は「同じ種類のもの」として扱い、中身に興味を持たない。if/switch の条件分岐を消せる。型チェックなしのダックタイピングでも多態性は成立する。
- **継承の真意は差分プログラミングではない**：継承の本質は「概念の一般化（is-a 関係）と抽象の分離」。コード再利用が目的なら委譲で十分。インターフェースは実処理を一切持てないため、差分プログラミングでは説明できない。
- **汎化**：具象の分析を先に行い、帰納的に抽象を発見するプロセス。先にひとりよがりな抽象を立てると役に立たない。
- **抽象は「機能する」**：構造化プログラミングの「抽象」は観念的に抽象とみなすだけ。一方 OOP の抽象は実際に機能し、上位構造が下位構造なしに成立できる。これが OOP と構造化プログラミングの決定的な違い。
- **構造化プログラミングとの関係**：構造化の機能分解（トップダウン）は上位が下位の正しさに依存し密結合。OOP はボトムアップ寄りで独立したオブジェクトを組み合わせる。構造化は OOP の前提となる基礎であり、否定されるものではない。

**気にした論点**
- 「多態性がなくても役立つ」の節：具象がひとつしかなくても継承が役立つという主張は理解できるが、「なぜその場合でも抽象に分けるのか」の動機づけ（開発工程のコントロール）が読み慣れないと見落としやすい。
- 「みなし抽象 vs 機能する抽象」：構造化プログラミングの「抽象」と OOP の「抽象」が同じ言葉で別の意味を指しており、読み分けが混乱しやすい。
- ParameterizedPet：「具象は間に合わせでもかまわない」節で唐突に登場し、何を示したいのかの接続が少し掴みにくい。

**結論**

この章は「オブジェクト指向とは何か」ではなく、**抽象（インターフェース）を使うことで上位構造を下位構造から独立させられる、これが OOP 固有のメリットだ**という話。

**残した保留**
- カプセル化・多態性・継承は「同時に現れる」と書かれているが、どう同時に現れるのかの具体イメージがまだぼんやりしている。
- 「抽象度が同じレベルのオブジェクトを返すメソッドにはデメテルの法則は適用されない」という判断を、実際のコードでどう見分けるのかがわかっていない。
- 構造化プログラミングの機能分解が「パッケージ境界の外に出たときだけ問題になる」という境界の引き方が、まだ腑に落ちていない。

== 実務の視点

**拾った主張**
- **カプセル化は「凝集度の最小単位」**。プロパティとメソッドが適切にそろっていること。足りなくて外から補う必要がある、あるいは肥大化しているのは失敗。→ 適用条件：責務の範囲が明確に言語化できている設計段階から意識する。
- **デメテルの法則 ＝「getter で取り出して連鎖させるな」**。`getEngine()->start()` は詳細の暴露。本来は `car->startEngine()` に委譲すべき。→ 適用条件：外部パッケージとの境界を越えるコードを書くとき。
- **多態性の実用は if/switch の撲滅**。NullLogger のサンプルが明快。フラグ変数をコンストラクタに渡す実装と、NullLogger を差し込む実装の比較は現場で使える説明材料になる。→ 適用条件：同じロジックに「切り替え」が生えてきたタイミング。
- **継承はコード再利用ではなく「抽象の確保」**。差分プログラミング目的の継承は委譲で代替すべき。インターフェースと実装の分離が本題。→ 適用条件：基底クラスの一般化概念が言葉で説明できるときのみ継承を使う。
- **具象が1つしかなくても抽象は意味がある**。ArticleRepositoryInterface のサンプル：具象がまだなくても上位パッケージを先に安定させられる。開発順序のコントロールが目的。→ 適用条件：データベースや UI の詳細が決まっていない初期フェーズ。
- **構造化プログラミングと OOP の境界**：パッケージ内の密結合は問題ない。問題になるのはパッケージ境界の外。内側はしっかり構造化、外と接する部分に OOP 的抽象を置く。

**使える場面**
- 「NullObject パターン」は今すぐ使える。テスト用モックの差し込みも同じ構造。フラグで分岐しているコードを見かけたら最初にこれを思い出す。
- 抽象インターフェースを先に書いてから DB や UI を後で詰める手順は、タスク並行進行として現実的。

**使えない場面・導入コスト**
- 「汎化は具象の分析から帰納的に発見せよ」という主張は正しいが、現場では最初から締め切りと実装圧力がある。具象を複数見るまで設計を待てないケースが多い。
- デメテルの法則を全員が理解していないチームに「メソッドチェーン禁止」だけ伝えると過剰適用が起きやすい（PSR-7 のような同一抽象 level 連鎖まで禁止する）。

**結論**

インターフェースで上位パッケージを先に安定させ、DB や UI 詳細の実装を後回しにできる設計手順が、自分の仕事の「詳細未確定なまま並行開発が必要なフェーズ」で直接効く。

**残した保留**
- **「良い抽象の見つけ方」が一番知りたいのに、「具象を分析してから帰納的に」で終わっている**。実際の業務でどのタイミング・何回の具象観察で汎化判断をするのか、基準が書かれていない。現場検証が必要。
- **トレードオフの定量的な記述がない**。インターフェースを増やすことのオーバーヘッド（ファイル数、読むコスト、チーム習熟コスト）と得られる安定性のバランスが、どんな規模・チームなら成り立つかが不明。
- **PHP のサンプルコードしかない**。他言語（Go, TypeScript, Python）での適用方法が省略されており、自分のチームの言語環境に翻訳できるか未確認。

== 批評の視点

**拾った主張（と疑問）**
- **「オブジェクト指向を定義することはできない」**
  → しかし直後に3つの特徴（カプセル化・多態性・継承）を列挙する。特徴を網羅的に挙げることは実質的に定義に近く、「定義できない」という宣言と矛盾する恐れがある。
- **「OOP と状態管理は関係ない」「OOP と手続き型は関係ない」**
  → Alan Kay の Smalltalk はオブジェクトの内部状態のカプセル化を中心概念に据えており、「OOP が状態管理に関心を持たない」は強すぎる。
- **「最小の抽象概念で素朴に作ったソフトウェアにするほうが、現実には高い拡張性を持つ」**
  → 反直感的な強い主張だが、根拠として具体的な事例も研究も示されていない。「現実には」という副詞が主張を強化しているが、証拠の代わりになっていない。
- **「構造化プログラミングと OOP の決定的な違いは、上位構造が下位構造なしに成立すること」**
  → C 言語の関数ポインタや不透明ポインタ（opaque pointer）でも、呼び出し側が実装詳細を知らずに動作は実現できる。「決定的な違い」という言葉が耐えられるかは疑わしい。
- **デメテルの法則において「抽象度が同じレベルのメソッドチェーンは問題ない」**
  → 「抽象度が同じ」という判断基準が主観的で定義されていない。「意図的かどうか」をどう判定するかの説明がない。

**論理の飛躍**
- 前提が明示されていない：「定義できない→3特徴で説明する」の移行が無断で行われている。「定義はないが特徴は言える」とは何が異なるのかが示されていない。
- ダックタイピングの節と「明確に定義されたインターフェースが重要」という記述の整合が取れていない箇所がある。
- 「犬猫サンプルを実務に置き換えられないなら、実務のサンプルを見ても応用できない」という論法は、サンプルの抽象化力に問題がある可能性を学習者の問題にすり替える構造になっている。
- 「抽象を先行させると役に立たない概念に縛られる」という主張は、DDD のようなアプローチと衝突するが、その反論への対応がない。

**反例で壊れる可能性**
- テンプレートメソッドパターンのように継承が本質的に意図された設計パターンでは「継承より委譲」の適用範囲が曖昧になる。
- Scala・F#・Kotlin 等の実際の言語設計で OOP と関数型が融合・干渉している事実と、「両者は軸が異なり干渉しない」という主張が衝突する。

**結論**

「継承の本質は差分プログラミングではなく抽象と具象の分離であり、それによって上位パッケージを安定させられる」という主張は、コードサンプルと安定依存の原則との対応によって最も具体的に支えられている。

**残した保留**
- 「OOP を定義できない」と「3つの特徴がある」の間の論理的な位置づけが未整理のまま章が進む。この曖昧さが残る限り、章全体の出発点の信頼性に疑問が残る。
- 「最小の抽象が最高の拡張性を持つ」は主張として魅力的だが、これを裏付ける実証的根拠が本章には存在しない。追加検証が必要。
- 「構造化プログラミングの抽象は観念的で機能しない」という断定は、関数ポインタ等の反例で覆る可能性があり、「決定的な違い」という表現の強度は検証待ち。
- デメテルの法則の適用判断における「抽象度の同値」基準が主観的であることは、実際のコードレビュー場面で運用可能かどうかの検証が必要。
