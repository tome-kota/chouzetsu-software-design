# 第2章 パッケージ原則

この章の中核は、パッケージの分け方・まとめ方・依存方向を REP・CRP・CCP・ADP・SDP・SAP の6原則でそろえ、変更コストを局所化することにある。1つの変更が1つのパッケージで完結し、依存がより安定した方向へ向く状態を設計目標として扱う。

## 章の主張

パッケージの分け方・まとめ方・依存の向けは、REP・CRP・CCP・ADP・SDP・SAP の6原則で決まる。1変更が1パッケージで完結し、依存が安定方向に向くことが、適切なパッケージ設計の状態である。

## 前提

- 「リリース」の定義がチーム内で統一されている（ライブラリ公開なのか、ブランチマージなのか、社内共有なのか）。
- パッケージの責務境界に関するチームの合意がある。
- 依存関係をコントロールできる状態にある（循環が既に蔓延していないこと）。
- ADP の議論は「意図しない循環が生じうる」という前提のもとに成立する。

## 実務上の効きどころ

- 新規モジュール・サービスの境界設計時のチェックリスト（CCP → CRP → ADP の順で確認）。
- コードレビューで「なぜこの変更が複数ファイルに波及するのか」を説明するための言語化。
- レガシーコードの品質評価軸として、循環依存の数を定量的に測定する起点。
- ブランチ運用があるチームでは、REP を「main マージ＝リリース単位」と読み替えて適用できる。

## 弱点と例外

- REP の「リリース」定義は、feature flag 運用・trunk-based development・継続的デプロイ環境では「マージ＝リリース」と一致せず、原則の射程が変わる。
- ADP の解決策「両者を1パッケージに閉じ込める」は、CRP の「責務に対して最小に」と衝突しうる。詰まると「まとめる」と言うなら、分割の意義が後退する。
- SAP の「抽象度の高さ」は4基準を束ねた広い概念であり、反例を排除しやすい定義になっている。チームでの判定は経験値依存になりやすい。
- 「分けすぎを後で混ぜるのは簡単」という前提に根拠がない。過度な分割は不要な依存・インターフェース設計の失敗・凝集度低下を招き、混ぜることも困難になりうる。
- 小規模プロジェクト（3〜5人規模）ではパッケージ分割自体より、責務合意と命名コストのほうが高くなる可能性がある。
- 循環依存の検出には静的解析ツールが実質必要で、ツール整備なしでは ADP を守っているかの確認手段が存在しない。

## 議論用の問い

1. SAP の「抽象度が高いかどうか」をチームで合意するための判定基準は作れるか。それとも結局レビュアーの経験値頼みになるか。
2. SDP を「真理」と呼んでいるが、これは設計規範（DIP など）と何が違うのか。事実の言語で語る必要があった理由は何か。
3. ADP で「循環が必要なら1パッケージに閉じ込める」と CRP の「詰め込みすぎるな」は、どう整合させるか。

== 理解の視点

**拾った主張**
- **REP（再利用・リリース等価の原則）**：リリースされたものだけを再利用すべきで、リリースと再利用の単位は一致させるべき。
- **パッケージ**とは公開ライブラリだけでなく、フォルダ分けされたコードの集まり全般を指す。Python や Node.js ではファイル単体もパッケージになりうる。
- **CRP（全再利用の原則）**：パッケージを利用する側は「全部使うか・全部使わないか」しか選べない。だからこそパッケージには多くを含みすぎてはいけない。責務は1つに絞るのが理想。
- 最初は分けすぎくらいでよい。後で混ぜるのは簡単だが、混ざったものを分け直すのは非常に困難。
- **CCP（閉鎖性共通の原則）**：1つの変更は、できるかぎり1つのパッケージだけを変えれば済む形にすべき。パッケージは「閉じている」べき。
- CRP と CCP は「分けるべき」と「まとめるべき」の両方から、責務とパッケージが 1:1 になることを説明している。凝集度が高いほど疎結合になりやすい。
- **ADP（非循環依存関係の原則）**：パッケージ間の依存に循環があってはならない。循環した2パッケージは実質1つの大きなパッケージと同じになる。どうしても循環が必要な要素は1つのパッケージに閉じ込めればよい。
- **SDP（安定依存の原則）**：パッケージの依存は常により安定したパッケージに向く。安定とは「変更の起きにくさ」。
- クリーンアーキテクチャでは依存は常に円の中心（ドメインモデル）に向かう。外側に行くほど不安定になる。
- **SAP（安定度・抽象度等価の原則）**：安定度が高いパッケージは抽象度も高くなければならない。抽象度の高さは 1) 抽象クラス・インターフェース 2) 詳細のないものだけに依存するロジック 3) 業務・技術に依存しない汎用概念 4) 言語標準と同等の業界標準、の4点で決まる。
- アーキテクチャ外側の DB やフレームワークが不安定なのではなく、自分のアーキテクチャ内の最外殻が不安定なのだという点に注意が必要。インフラ層の役割はそのギャップを埋めること。

**気にした論点**
- 「ドメインモデルはもっとも再利用性が高い」という箇所が「直感に反する」と書かれているが、なぜそうなるかの説明が短く、腑に落ちきらなかった。
- SDP の「向けるべき」ではなく「常に向く」という表現が観察上の事実として語られているのか、規範として語られているのかが途中で混乱した。
- SAP の「抽象度の高さ」を決める4点が並んでいるが、それぞれの境界がやや曖昧だった。

**結論**

この章は、パッケージをどう分け・どうまとめ・どう依存させるかを REP・CRP・CCP・ADP・SDP・SAP の6原則で整理した話。

**残した保留**
- REP で「適切な粒度」が大事と言われているが、その見極め方はまだ具体的にわからない。
- SDP の「抽象度が高いほど安定度が高い」は相関として述べられているが、因果関係なのか単なる傾向なのかがはっきりしない。
- インフラ層の構造（内部と外部の両方に依存する）が本文だけではイメージしきれていない。

== 実務の視点

**拾った主張**
- **REP（リリース等価）**：チーム開発では「main ブランチへのマージ＝リリース」と読み替えられる。作りかけコードを共有フォルダに混ぜたまま並行開発するのが問題の根本という指摘は現場の実感と合う。
- **CRP（全再利用）**：「1パッケージに詰め込みすぎるな」。全部入りフレームワークより分離ライブラリのほうが交換コストが低い。
- **CCP（閉鎖性共通）**：「1つの機能変更で1パッケージしか触らなくて済む分け方が正解」。連鎖変更コストを最小化するための原則。
- **ADP（非循環依存）**：循環依存が必要になったら「2つを同一パッケージに閉じ込める」が解決策。
- **SDP/SAP**：「自身より不安定なものに依存した時点で自身も同等に不安定になる」は原理的な制約として理解しておく価値がある。

**使える場面**
- 新規モジュール・サービスの境界設計時のチェックリスト（CCP・CRP・ADP 順に確認）。
- コードレビューで「なぜこの変更が5ファイルに触れるのか」を説明するときの言語として使える。
- レガシー品質の評価軸として「循環依存がいくつあるか」は定量的に話せる。

**使えない場面・制約**
- 既存コードに循環依存が山積みの現場では、原則の理解よりも解消の手順が先に必要。
- 「最初は少し分けすぎでいい」という助言は正しいが、初期の分割粒度の目安がない。
- SAP は概念が重なり合っていて「抽象度が高いかどうか」の判断を属人依存にしやすい。
- 循環依存を検出するツール導入（静的解析）がないと ADP はモニタリングできない。

**結論**

「1機能変更が何ファイルに波及するか」を設計判断の基準にすれば、CRP・CCP の原則は現場の設計レビューでそのまま使える。

**残した保留**
- SAP の抽象度判定基準が不明。結局レビュアーの経験値頼みにならないか。
- 小規模プロジェクト（3〜5人）で同じ粒度の分割コストが見合うか未確認。
- 「混ざったものを分け直すのは非常に困難」とあるが、段階的リファクタリングの順序については記述なし。

== 批評の視点

**拾った主張（と疑問）**
- **「ドメインモデルはもっとも再利用性の高い要素」**：プロジェクト内での共有を「再利用」と定義し直すことで成立させているが、この定義の切り替えが明示されない。
- **「master マージがリリース」**：feature flag 運用・trunk-based development・継続的デプロイの環境ではマージ＝リリースとは一致しない。
- **「最初はちょっと分けすぎぐらいで十分」**：直前に「変更想定しすぎて複雑さを持ち込むな」と書いてある。この2文は方向性が矛盾しており、優先基準が与えられていない。
- **「パッケージは常により安定したパッケージに依存する（真理）」**：設計ガイドラインであって観測的事実ではない。断言の根拠は定義の同語反復に近い。
- **「抽象度イコール安定度」**：相関の証拠が提示されていない。反例を排除しやすい定義に拡張されている。
- **「分けてあるものを混ぜるのは簡単」**：根拠がない。過度な分割は不要な依存増加・インターフェース設計の失敗・凝集度低下を招く。

**論理の飛躍**
- ADP の解決策「両者を1つのパッケージに閉じ込める」は CRP の「責務に対して最小に」と衝突しうる。
- SAP で psr/log を安定の例として挙げているが根拠は「多くが使っているから」。合意による安定であり、原則から演繹された安定ではない。
- 「プログラミング言語は究極の抽象」：Python 2→3 の破壊的変更など、言語自体が変動する現実がある。

**結論**

この章の主張のうち証拠と論理が十分に支えられているのは、「循環依存するパッケージは実質的に単一の大きなかたまりと等価になる」（ADP）という部分のみ。

**残した保留**
- REP の「リリース」定義がチーム・組織によって異なる場合、原則の適用範囲はどこまでか。
- SAP の抽象度4基準が「抽象度」という単一の軸に還元できるという主張には追加の論証が必要。
- SDP を「真理」と呼ぶことの正当性。DIP など他の設計規範と何が違うのか説明されていない。
