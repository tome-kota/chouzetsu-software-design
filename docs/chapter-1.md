# 第1章 クリーンアーキテクチャ

この章の中核は、凝集度を高め、安定度の高いものへ依存方向をそろえることで、変更コストを下げるという発想にある。ここでいうアーキテクチャの価値は、プログラムを動かすことそのものではなく、ソフトウェアを壊さずに育て続けるための開発パフォーマンスにある。

## 章の主張

凝集度を高め、安定度の高いもの、つまり本質に近いドメインモデルへ向かって依存方向を統一することで、変更コストを下げられる。クリーンアーキテクチャはそのための固定テンプレートではなく、依存方向と責務配置を整えるための設計パターンとして理解するのがよい。

## 前提

- 対象は「役立っているソフトウェアを壊さずに成長させ続ける」状況に限られる。使い捨て、短命、小規模のソフトウェアでは投資対効果が合わない。
- ドメインモデルが「安定できるもの」として識別可能であることが前提になる。ドメイン自体が流動的なら、この前提は崩れる。
- 依存の逆転を実装で成立させるには、オブジェクト指向の技法を別途理解している必要がある。本章単体では実装までは完結しない。

## 実務上の効きどころ

- 設計レビューやコードレビューで、なぜその構造が触りにくいのかを、凝集度、依存方向、安定度の3つの語彙で説明できる。
- 汚い設計の実害を「変更が分散する」「同一箇所に意味の異なる責務が混在する」と切り分けて話せるため、指摘の解像度が上がる。
- 「本質への変更は重い」という視点があると、変更コストをステークホルダーに説明しやすくなる。
- アプリケーション全体を一度に作り替える必要はなく、概念パターンとして局所適用できる。

## 弱点と例外

- 「アーキテクチャはプログラムの動作に貢献しない」という言い切りは強すぎる。可用性やレイテンシに直結する設計では、構造選択が動作特性に直接影響する。
- 「本質とは変わりにくいもの」という説明は循環的で、何が本質かの識別基準として弱い。
- 良いアーキテクチャの採用コスト、つまり設計、学習、議論の負荷が十分に扱われていない。
- 4層構造を強く説明した後に「固定形ではない」と後退するため、読者がどこまで型として受け取るべきかが曖昧に残る。
- チームのドメイン理解が浅い段階では、内側に置いたはずのドメインモデルの方が先に揺れる逆転現象が起こりうる。

## 議論用の問い

1. 何が本質で何が非本質かは、誰がどのプロセスで決めるべきか。
2. 凝集度と結合度は、現場では本当に独立して評価できるのか。
3. 設計コストまで含めたとき、クリーンアーキテクチャの採用が割に合う境界はどこか。

== 理解の視点

**拾った主張**
- **アーキテクチャ**とは「何をどこに配置するかを決めたもの」。クリーンアーキテクチャの「クリーン」は構造上の実体ではなく設計指針の名前。
- アーキテクチャの良し悪しはプログラムの動作に直接貢献しない。貢献するのは「ソフトウェアを開発するパフォーマンス」。
- 汚い設計の実害は2つ：①まとまりが悪くあちこち変更が必要になる、②同じ箇所に意味の違うものが混ざって壊れやすい。
- **凝集度**とは「関係の強いもの同士だけが集まっていること」。**疎結合**とは無駄な癒着がなくつながりが切れている状態。密結合になると凝集度は下がる。
- **安定度**とは「コードの変わりにくさ」。動作の安定ではなく変更頻度の話。本質か非本質かという視点が予測に使える。
- 4層構造（内→外）：**ドメインモデル**→**ユースケース**→**インターフェースアダプター**→**インフラストラクチャ**。依存の向きは常に外から内へ。
- クリーンアーキテクチャは4層という固定の型ではなく、概念のパターン。オニオン・ヘキサゴナルアーキテクチャと共通コンセプトを持つ。

**気にした論点**
- 凝集度と結合度は別概念と言いながら相関があるとも書いており、違いの境界が掴みにくい。
- 3層目と4層目の依存逆転の話が「後の章で説明する」で終わっており、仕組みが全くイメージできない。
- 「安定度」という用語が直感と逆（動作の安定ではない）なので混乱しやすい。

**結論**

この章は「なぜきれいな設計が必要か」と「クリーンアーキテクチャとは何か」を、凝集度・依存・安定度という3つの概念を軸に説明した話。

**残した保留**
- 依存の逆転の仕組みが具体的にどういうことか。「オブジェクト指向」で解消されると書かれているが、現時点では仕組みが全くイメージできていない。
- ドメインモデルを「何にも依存しない」よう書くとは、実際のコードでどういう状態を指すのか。

== 実務の視点

**拾った主張**
- アーキテクチャは開発パフォーマンスへの貢献である。適用条件：「役立っているソフトウェアを壊さずに成長させ続ける」状況に限り有効。使い捨て・小規模なら不要。
- 汚い設計の実害2点（変更の分散・意味の混在）は現場で感覚的に「なんかつらい」と言っていたことに名前をつけてくれる。
- 安定度の予測に「本質か非本質か」の視点が使える。本質への変更依頼はステークホルダーにも覚悟があり、変更コストへの理解を得やすい。
- クリーンアーキテクチャは「どこにでも部分適用できる」概念パターン。アプリ全体を型にはめるフレームワークではない。

**気にした論点**
- 「なんかこのコード触りにくい」という感覚を設計レビューで言語化する場面で直接使える。
- ドメインモデルの洗練に業務ドメイン理解と相当の工数がかかる。チームのドメイン理解が薄い段階では本末転倒になりうる。
- 依存の逆転の実装技法が後章委ねで実装イメージが付かない。概念だけで試みると誤った依存構造を「クリーンアーキテクチャのつもり」で作るリスクがある。
- 従来の「凝集度・疎結合」にクリーンアーキテクチャが上乗せしているのは「依存方向の安定度」という視点。

**結論**

設計レビュー・コードレビューで「なぜこの構造がまずいか」を人に説明する根拠として、凝集度・依存方向・安定度の3概念がそのまま使える。

**残した保留**
- ドメインモデルの洗練にどれだけの反復が必要か。スプリント何本分を見込むべきかが示されていない。
- 既存のスパゲティコードへの後付け適用（リファクタリングでの段階的導入）の進め方が本章では触れられていない。
- 依存逆転の実装技法が後章任せのため、現時点では「採用できるか」の最終判断を保留。

== 批評の視点

**拾った主張**
1. 「クリーンアーキテクチャは過去に語られた何よりも根源的な問題をシンプルに明らかにする」→ 比較対象が示されておらず、「新しさ＝根源性」の因果関係に根拠がない。
2. 「アーキテクチャの良し悪しはプログラムの動作に直接貢献しない」→ フォールトトレラントやレイテンシ直結のデータ配置など、動作への影響がある反例が存在する。
3. 「本質は一度安定すればなかなか変わらない」→「安定すれば」という条件節が「安定にできるはずだ」という主張とセットになっており論証になっていない。
4. 「安定度の予測に本質か非本質かという視点が使える」→「本質＝変わりにくいもの」の定義が循環しており、判断基準として機能しない場面がある。
5. 4層構造を章の大部分で説明しながら「決まった形ではない」と後退。どちらを拠り所にすべきかが不明。
6. 「依存方向の自由な制御にオブジェクト指向が役立つ」→ 核心的主張として提示されているが証明が先送りで、本章の範囲では未証明の宣言に留まる。

**気にした論点**
- 「良いアーキテクチャ→開発パフォーマンス向上」は成り立ちそうだが、採用コスト（設計・学習・議論）が考慮されていない。
- 「密結合になると凝集度は低くなる」という図式に本文自身が留保をつけているのに、後の説明では裏表のように扱われる。
- 安定度分布と本質/非本質の一致は観察的事実として述べられているが、実例も数量的根拠も示されていない。

**結論**

「凝集度を高め、安定度の高いものに依存する方向に設計することで変更コストが下がる」という主張は一定の論理的整合性を持つが、いくつかの前提が未検証のまま。

**残した保留**
- 「本質」の識別可能性：急成長領域や要求が流動的なスタートアップでは主張全体の前提が崩れる可能性がある。
- 依存逆転の実現可能性：後の章が不十分であれば、この章の説得力が事後的に低下する。
- 効果の測定問題：同一要件をクリーンと非クリーンで開発した比較データがなく、主張の反証可能性が低い。
